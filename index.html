Engineering Challenge 1
The point cloud data was too large — causing pipeline lag

Raw CS20 point cloud data per frame was large enough to introduce measurable lag when passed through a GStreamer buffer on every frame. The solution was a custom bit-packing algorithm.

Each 3D point (px, py, pz) is encoded as a compact fixed-width bit-stream: projected X into x_bits, projected Y into y_bits, depth Z into 14 bits. No heap allocation in the per-frame data path. Buffer sizes dropped significantly and the lag was eliminated.
Engineering Challenge 2
The naive distance measurement approach fails in practice

Taking the mean depth of all LiDAR points inside a bounding box produces unreliable results for two reasons: bounding boxes include background and edge noise, and camera + LiDAR are not hardware-synchronised on an embedded system.

Solution — two-step refinement:

Step 1: Shrink the bounding box to 20% of its original size from the centre. This discards noisy perimeter pixels and focuses the estimate on the stable core of the person's body.

Step 2: Sample depth values only along the diagonal of the reduced box — not the full area. A diagonal sample is far less sensitive to frame-to-frame synchronisation jitter and requires significantly less computation than a full area mean.

Result: stable, accurate distance measurements under real operating conditions — asynchronous streams, partial occlusions, varying lighting.

Architecture
Generic LiDAR abstraction — designed for the future

Instead of hardcoding CS20 support, the internals were redesigned around a C++ abstraction layer. The GStreamer plugin (pure C) never knows which LiDAR is active — it calls five bridge functions only.

GStreamer Plugin (pure C)
        │
        │  calls only 5 bridge functions
        ▼
   C Bridge Header          ← explicit C ↔ C++ boundary
        │
        ▼
   C++ Wrapper Layer
        ├── LidarBase (abstract interface)
        ├── CS20 (Synexens 3D)   — compiled if -Dlidar_type=CS20
        ├── G2  (YDLiDAR 2D)    — compiled if -Dlidar_type=G2
        └── Wrapper Class        — owns the active backend

Adding support for a new LiDAR sensor requires implementing one derived class and one overlay handler — zero changes to the plugin itself. This is the Open/Closed Principle applied to embedded hardware integration.

Build System
Build-time sensor selection with Meson

Passing -Dlidar_type=CS20 at configure time pulls in only the CS20 SDK and its implementation. The final binary contains no dead code from unused sensor backends — keeping executable size minimal on constrained embedded hardware. Platform independent: same codebase runs on Jetson Orin Nano and Xilinx ZCU106 without modification.

Engineering Challenge
Transmitting LiDAR data inside an H.264 stream — the SEI approach

The requirement was to transmit both camera video and LiDAR depth data to a remote receiver over UDP as a single stream — no separate channel, no receiver-side sync complexity.

H.264 and H.265 define SEI (Supplemental Enhancement Information) NAL units — side-channel data that travels inside the bitstream. Standard decoders skip SEI they don't understand, leaving the video stream fully intact. LiDAR data rides alongside the video invisibly.

Understanding the SEI NAL structure required reading the H.264/H.265 specifications: 4-byte start code, NAL unit header (different for AVC vs HEVC), payload type byte 0x05, multi-byte size encoding with 0xFF prefix bytes for payloads over 255 bytes, 16-byte UUID, payload, RBSP stop bit. Figured out from scratch using only SDK documentation and the NAL specification.

Engineering Challenge
The DEADBEEF escape algorithm — preventing byte corruption

Embedding arbitrary binary data inside H.264 introduced a critical problem: if the LiDAR payload contained byte sequences matching H.264 start codes (0x00 0x00 0x01 and variants), the decoder would misinterpret payload bytes as NAL unit boundaries — corrupting the entire stream with garbled video and incorrect depth data.

DEADBEEF escape algorithm:

Encoder: scan LiDAR payload for 0x00 0x00 followed by 0x00/0x01/0x02/0x03. Replace with 0x00 0x00 0xDE 0xAD 0xBE 0xEF XX — the DEADBEEF marker cannot appear in a legitimate H.264 start code context.

Decoder: scan extracted payload for the marker pattern, restore original 0x00 0x00 XX sequence. Runs in-place with no additional allocation.

Result: no stream corruption, no memory pressure, minimal per-frame CPU overhead. The entire algorithm is a tight linear scan of the payload buffer.
